## 1. 背景：A–C 阶段简述

### 1.1 Phase A：基线冻结与数据准备

Phase A 建立了稳定的评测流水线（StrategyQA、GSM8K）和数据分层体系。
- 冻结基础模型在 StrategyQA freeform 任务上约 76% 精度
- 完整 CoT 风格 GSM8K PEFT 会退化；较短目标风格可改善
- 关键输出：标准化 artifact 体系（run directory、manifest、metrics.json）

### 1.2 Phase B：PEFT 基线

Phase B 验证了 PEFT 对 StrategyQA 的一致增益，同时发现跨任务干扰。

核心结论：
- `B1_FIRST`：StrategyQA freeform 验证集 = 0.7617（+0.07 vs 冻结）
- GSM8K 长 CoT 出现跨任务干扰；仅短目标格式可提升
- 结论：纯微调已无法提供差异化研究贡献，需要**过程感知控制**

### 1.3 Phase C：BCR/ABR 原型

Phase C 构建了三个核心组件：
1. **前缀级价值头**：MLP 接骨干隐状态，估计从当前前缀出发的成功概率
2. **BCR-lite**：Bellman 一致性正则化训练路径（token 级 V(h_t) ≈ r_t + γ·V(h_{t+1})）
3. **ABR-lite**：步骤级自适应验证控制器，路由 gen/ver/fin 三种动作

Phase C 结论（2026-03-10 重解释）：
- 工程管道稳定（C1/C2 协议、质量诊断）
- **关键问题**：StrategyQA 缺少公开 PRM 级步骤质量标注，不再适合作为价值头验证主基准
- BCR/ABR 的"价值平滑性"假设不能替代高质量过程监督

## 2. Phase D：外部 PRM 辅助价值学习

**时间**：2026-02-20 ~ 2026-03-10
**研究假设**：在有丰富步骤标注的数学推理数据（Math-Shepherd、PRM800K）上预训练价值头，能否迁移回 StrategyQA？

### 2.1 Phase D 关键实验

| 运行 | 数据源 | 主要结果 | 含义 |
|------|--------|---------|------|
| DT2_MATH_SHEPHERD_SEED3_STABLE | Math-Shepherd 高质量三元组 | pair_acc 高，排序可学习 | 外部数据可学习 |
| DT2_MS_SEED3_C1_TRANSFER | Math-Shepherd → StrategyQA | **接近随机** | **迁移失败** |
| DT3_PRM800K_SEED3_STABLE | PRM800K | 稳定弱基线 | 可用但较弱 |
| DB3_SQA_BRIDGE_FULL_RANK | MS 预训练 + SQA 排序持续训练 | SQA 排序改善 | 桥接有一定效果 |
| DB4_SQA_BRIDGE_FULL_JOINT | 标量/联合校准目标 | **排序主动受损** | 校准目标有害排序 |

### 2.2 Phase D 核心发现

1. **排序分支可学习**：在 Math-Shepherd 高质量三元组上，排序分支确实可学习
2. **源-目标迁移失败**：`DT2_MATH_SHEPHERD → StrategyQA` 接近随机，源-目标域差距真实存在
3. **联合校准目标有害**：DB4 中即使 Brier 损失下降，排序也会主动受损——目标设计优先级：过程排序 > 标量校准
4. **战略转向**：外部可学习性 ≠ 目标迁移；StrategyQA 不再作为主基准

### 2.3 数据语义修复（一级风险，2026-03-10）

**问题**：原始 Math-Shepherd / RLHFlow / PRM800K 的 `+/-` 标签通过最近邻负样本转换器构建对，从单条轨迹中混合了进展信号与局部错误信号。

**影响**：构建的"step pair"实际混合了不同语义，早期模型实际上在学习"哪个步骤更长"或"哪个步骤位置更深"，而非真正的局部错误检测。

**修复**：
- 默认 `step_label_pair_mode` 改为 `first_bad_edge_strict`
- 只保留：第一个负步骤之前最后干净前缀 vs 包含第一个坏步骤的前缀
- 第一个负步骤前无正步骤的样本直接丢弃
- 所有修复前 artifact 降级为 `legacy`

**文档**：`docs/data_semantics_risk_audit_20260310.md`

## 3. Phase E 启动：方法论修正与基准重设计

**触发条件**：Phase D 证明过程排序可学习，但跨数据集迁移结构性困难。StrategyQA 缺乏 PRM 级标注。

### 3.1 Phase E 核心设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 评测指标 | **ProcessBench F1**（MATH + GSM8K 子集） | 官方标准：F1=2×Acc_err×Acc_cor/(Acc_err+Acc_cor) |
| 骨干网络 | **Qwen2.5-Math-PRM-7B（冻结）** | 专为数学过程验证训练，显著优于 Instruct 骨干 |
| 可训练组件 | **MLP 价值头**（512-dim → 1 标量） | 轻量，无需大量计算，可复现 |
| 数据来源 | DPO 数学步骤对 + Math-Shepherd strict | 两类互补：DPO 提供偏好对比，MS 提供过程错误定位 |
| 主基准 | ProcessBench（MATH 1000题 + GSM8K 1000题） | 跨分布泛化能力的真实评测 |

### 3.2 ProcessBench F1 指标解释

**官方 F1**：
```
F1 = 2 × Acc_error × Acc_correct / (Acc_error + Acc_correct)
```
- **Acc_error**：在有错误的推理轨迹中，正确检测到错误的比例（阈值以下）
- **Acc_correct**：在全部正确的推理轨迹中，不产生误报的比例（全程高于阈值）
- 决策规则：找到第一个 score < τ 的步骤（τ 在分数范围内自动调优）

**重要意义**：F1 平衡了错误检测率和误报率，比单纯的 AUC 更能反映实际可用性。

### 3.3 社区基线（2026-03-12 更新）

| 模型 | GSM8K F1 | MATH F1 | 训练方式 |
|------|---------|---------|---------|
| Qwen2.5-Math-7B-PRM800K | 68.2 | 62.6 | PRM800K 全量微调 |
| Skywork-PRM-7B | 70.8 | 53.6 | GSM 过拟合 |
| **Qwen2.5-Math-PRM-7B** | **~73.5** | **~73.5** | **7B 目标** |
| EDU-PRM-7B | 95.0 | 88.2 | 熵步骤边界 + 严格数据 |
| **我们 PBR26** | **0.768** | **0.686** | **当前最优（冻结骨干）** |
| **我们 PBR32** | 0.776 | **0.689** | **当前最优（LoRA）** |

## 6. 反模式诊断（Anti-Pattern A–G）

这是 Phase E 中最重要的诊断工作，历时约一周（2026-03-11 集中）。

### 6.1 反模式 A：MS 数据几何/长度偏置（**确认根因**）

**现象**：使用 Math-Shepherd fanout/grid 对训练，ProcessBench MATH AUC 接近 0.489（无正迁移）。

**根因诊断**：
- MS fanout/grid 对中，rejected（坏）序列比 chosen（好）序列长约 **+194/+203 tokens**
- ProcessBench 中，erroneous（有错误）前缀往往**更长**（更多步骤才出错）
- 训练信号与测试分布方向**相反**，模型学到"更长=更坏"，在 ProcessBench 上则反转

**对照实验**：
| 数据 | MATH F1 | AUC | 含义 |
|------|---------|-----|------|
| 纯 DPO（无 MS fanout/grid）| 0.721 | — | DPO 数据是正确迁移的最低要求 |
| MS strict only（无 fanout/grid）| ~0.489 | — | MS 单独不够 |
| DPO + MS strict | **0.644+** | — | 两者互补 |

**修复**：
1. 只使用 `ms_v2_strict`（first-bad-edge 模式，已去除 fanout/grid）
2. 确保 DPO 数据始终存在

### 6.2 反模式 B：terminal 比例失控

**现象**：
- terminal_top1 ≈ 0.05：40–48% 的全正确样例未被监督（terminal 信号严重不足）
- 强制增大 terminal 比例：局部 AUC 下降，精度换回的代价过大

**发现**：gated_mlp 架构可避免分数反转，但无法解决 local/terminal 权衡。

**结论**：terminal anchor 是必要的，但适量（PRX1 level）即可；不应将 terminal 比例最大化。

### 6.3 反模式 C：置信度阈值过高

**问题**：原始 min_conf=0.70 过滤太激进，丢失大量低置信度但有效的对。

**修复**：默认降至 **min_conf=0.55**。

### 6.4 反模式 D：评测指标缺失

**问题**：早期代码用 AUC 作为主指标，ProcessBench 官方使用 F1。AUC 高的模型 F1 不一定高（阈值设置问题）。

**修复**：实现 `compute_processbench_f1()` 函数（`src/ours/phase_e/benchmark_eval.py`）。
官方 F1 = 2×Acc_error×Acc_correct / (Acc_error + Acc_correct)，τ 在全分数范围内自动调优。

### 6.5 反模式 E：dual_head 路由过旋转

见 §5.2 PRX2 结果。dual_head 实现正确，但结论为不推荐作为主线。

### 6.6 反模式 F：R-PRM 格式

compact_verdict 格式确认正确（0% 风险），早期担忧的 token 截断诊断为 MODERATE（1024 token 以上 11.6% 隐藏差异），但 compact_verdict 已修复。

### 6.7 反模式 G（最危险）：灾难性 logit+semantic 配方

**现象**：`ranking_target_space=logit` + `confidence_semantic` + `ranking_score` 在混合 terminal 数据上触发训练崩溃，MATH F1 < 0.40。

**机制**：三个不兼容项同时启用导致梯度方向不一致，loss 爆炸或 collapse。

**修复**：`src/ours/phase_e/recipe_safety.py` 实现 `ANTI_PATTERN_G_FULL` 块，在骨干加载前阻止该配方，防止静默污染未来实验。

**文档**：`docs/antipattern_fix_experiments_20260311.md`

## 7. Phase E 第三轮：多源数学 + Terminal BCE（PBR13–PBR27）

**时间**：2026-03-11
**核心贡献**：Terminal BCE 特性（BiRM-lite）将 MATH F1 从 0.644 提升至 0.686（+4.2 分）

### 7.1 Terminal BCE 实现（BiRM-lite）

**动机**：终端步骤（全正确或全错误序列末步）提供明确标签（正确→1，错误→0），可用 BCE 损失直接强化。

**实现**：
- 文件：`src/ours/phase_e/training.py:1309`（`_compute_single_pair_objective`）
- 新参数：`--terminal-bce-lambda`（默认 0.0，推荐 0.25）
- 逻辑：在对比对目标上额外施加 BCE(chosen→1, rejected→0)，由 terminal 路由权重门控

### 7.2 PBR19 和 PBR26：冻结骨干最佳结果

| 运行 | 数据 | Terminal BCE λ | 训练周期 | MATH F1 | GSM F1 |
|------|------|---------------|---------|---------|--------|
| PBR12（基线） | DPO+MS strict | 0 | — | 0.644 | 0.752 |
| PBR19 | DPO+MS full | **0.25** | 10ep | **0.683** | **0.778** |
| PBR26 | DPO+MS full | **0.25** | 5ep | **0.686** | 0.768 |
| PBR27_mlp1024 | DPO+MS full | 0.25 | — | 0.666 | 0.784 |

**Terminal BCE 是关键突破**：λ=0.25 将 MATH F1 从 0.644 → 0.683–0.686（+4 分）。

**PBR19 vs PBR26**：
- PBR19（10ep）：MATH F1=0.683，GSM F1=**0.778**（GSM 最优）
- PBR26（5ep）：MATH F1=**0.686**，GSM F1=0.768（MATH 最优）
- PBR19 在 GSM 上轻微过拟合，PBR26 反而更好——5ep 是更优选择

**MLP 容量实验**（PBR27_mlp1024）：隐层 512 → 1024 使 MATH F1 下降（0.666），更大容量无益，反而可能过拟合。

### 7.3 DPO vs DPO+MS 系统对比

| 数据 | MATH F1 | GSM F1 | 教训 |
|------|---------|--------|------|
| 仅 DPO（2398 对）| 0.622 | 0.735 | DPO 单独不够 |
| DPO + MS strict（5705 对）| 0.644 | 0.752 | MS 补充局部错误信号 |
| DPO + MS full（~7366 对）| **0.686** | **0.778** | MS full 含更多信息（但含噪声） |

两类数据互补：DPO 提供偏好对比语义，MS 提供过程错误定位语义。

### 7.4 Instruct vs Math-PRM 骨干对比

| 骨干类型 | 实验 | MATH F1 | GSM F1 |
|---------|------|---------|--------|
| **Qwen2.5-Math-PRM-7B** | PBR26 | **0.686** | **0.778** |
| Qwen2.5-7B-Instruct | FLB2B ta_sweep r000 | 0.141 | 0.183 |
| Qwen2.5-7B-Instruct | FLB2B ta_sweep r020 | 0.236 | 0.258 |
| Qwen2.5-7B-Instruct | FLB2B s7 gated_mlp | 0.381 | 0.438 |

**结论**：Instruct 骨干完全不适合 ProcessBench 验证任务，差距约 0.30–0.45 MATH F1 分。Math-PRM 骨干是强制要求。

### 7.5 冻结骨干天花板确认（2026-03-11 末）

经过 PBR1–PBR27 全系列，冻结骨干+MLP 实验平台确认：

> **冻结骨干天花板：MATH F1 ≈ 0.683–0.686**

距社区目标（Qwen2.5-Math-PRM-7B 完整微调 ~0.735）约 **5 F1 分**。
假设：骨干特征表示未充分适配 ProcessBench 分布，骨干自适应（LoRA）可能是突破方向。

## 8. Phase E 第四轮：LoRA 引入（PBR28–PBR32）

**时间**：2026-03-11 ~ 2026-03-12
**动机**：冻结骨干天花板约 0.686，需骨干自适应突破

### 8.1 LoRA 技术实现

**新增代码组件**：

### 10.2 GRPO 奖励整形：Clip+Delta

**来源**：Zeng et al. 2024（arXiv:2410.15115）

**问题**：原始 PRM 均分作为 RL 奖励导致奖励欺骗（模型产生大量微小步骤累积分数）。

**修复公式**：
```python
# Phase F reward shaping: Clip+Delta
r = mean(clip(PRM(t) - PRM(t-1), -0.3, +0.3))
```

**实现**：`scripts/phase_f_grpo_lite.py`，`PRMScorer.score_clip_delta()` 方法，`--reward-shaping clip_delta`（默认）。

### 10.3 Best-of-N（BoN K=4）实验

**设置**：Qwen2.5-Math-7B-Instruct 生成器，k=4，200 GSM8K 问题，seed=42

| 评分器 | greedy | random@4 | prm@4 | Δ vs greedy | Δ vs random | phase_f3 |
|--------|--------|----------|-------|-------------|-------------|---------|
| **PBR32（LoRA r=8 all-28）** | 0.905 | 0.900 | **0.920** | **+1.5%** | **+2.0%** | **PASS** |
| PBR26（冻结）v4 r2 | 0.910 | 0.915 | 0.925 | +1.5% | +1.0% | **FAIL** |

**Phase_f3 门控规则**：`prm_vs_random_delta ≥ 0.02`（≥2% 改善）

**PBR26 为何 FAIL（详细分析）**：
1. 生成器足够强（91% greedy），random@4 = 91.5%（已超过 greedy）
2. PRM 需将 prm@4 从 91.5% 提升至 ≥93.5% 才能通过
3. 实际 prm@4 = 92.5%，差 1 个百分点

**PBR32 为何 PASS**：
1. random@4 = 90.0%（低于 greedy），说明解之间差异度更大，PRM 辨别空间更大
2. prm@4 = 92.0%，prm_vs_random = +2.0%（恰好通过）
3. **LoRA 训练即使仅提升 ProcessBench F1 +0.003，也显著提高了 BoN 辨别力**

**额外分析**：PBR26 的 `filler_outrank_rate=0.208`（高）——填充词（filler tokens）系统性地被评为高分，导致 PRM 无法有效区分"好解"和"填充词堆砌解"。

### 10.4 GRPO 全系列实验

#### 设置
- 生成器：Qwen2.5-Math-7B-Instruct
- 任务：GSM8K（200 训练题）
- 预训练精度：**95.5%**

#### 完整结果表

| 运行 | PRM | 奖励整形 | pre-acc | post-acc | Δ | gate | 根因 |
|------|-----|---------|---------|----------|---|------|------|
| v1（PBR32） | LoRA | raw_score | 0.950 | — | — | CRASH | 磁盘满（5×43GB=215GB）|
| **v2 outcome-only** | 无 | outcome | **0.955** | **0.950** | **-0.005** | **FAIL** | 组内奖励方差=0 |

#### GSM8K 天花板问题（根因分析）

```
生成器 pre-training accuracy = 95.5% on GSM8K
    ↓
大多数 k=4 解方案组：全部正确（outcome reward = [1,1,1,1]）
    ↓
组内 advantage = 0（mean=1, std=0）
    ↓
GRPO 更新 = 纯噪声（或轻微退化）
    ↓

    ↓
GRPO FAIL（所有5次实验）
```

**Clip+Delta 的作用**：v4 r2 使用 PBR26 frozen + Clip+Delta，post-acc = pre-acc（0.955，无退化），说明 Clip+Delta 至少阻止了 v2 中 outcome-only 的轻微退化。但无法突破饱和天花板。

**未来 GRPO 必要条件**：使用生成器基线 ~70–80% 的更难基准（MATH500 ~45%，AMC ~20%，AIME ~15%）。

### 10.5 ABR RL 控制器全面扫描

**研究问题**：ABR 步骤级路由器是否可以通过 RL 优于启发式阈值？

**实验设置**：
- 基于 PBR19/PBR32 评分轨迹的离线 RL 训练
- 10 个控制器变体：线性、MLP-64、MLP-128、GRU-32、BC warmstart、REINFORCE
- 目标：最小化验证步骤同时最大化错误检测 F1

#### MATH 域结果（定论）

| 方法 | MATH F1 | 备注 |
|------|---------|------|
| **启发式阈值 τ=0.38** | **0.822** | **最优** |
| 最优 RL 变体（BC warmstart MLP）| 0.779 | 比启发式差 0.043 |
| 所有其他 RL 变体 | <0.779 | 启发式全胜（10/10）|

**根因**：MATH 轨迹中 **84.3% 的错误样例**包含分数振荡（平均每例 1.42 次振荡，std=0.219）。RL 策略无法从这些噪声振荡模式中学到稳定规律。启发式"任何下降即触发"（τ=0.38）达到 86.5% 错误检测率，接近最优。

#### GSM8K 域：种子偏差问题暴露

| 种子 | RL F1 | 启发式 F1 | Δ | 样本数 |
|------|-------|---------|---|------|
| 42（×15 配置，早期结论）| 0.837 | 0.793 | **+0.044（RL 胜）** | 60 样例 |
| 1（×2 配置）| 0.839 | **0.916** | -0.077（启发式胜）| 60 样例 |
| 7（×2 配置）| 0.796 | **0.900** | -0.104（启发式胜）| 60 样例 |
| 123（×1 配置）| 0.787 | **0.903** | -0.116（启发式胜）| 60 样例 |

**种子 42 是统计异常**：该特定 60 样例测试分割正好包含大量阈值附近的样例，启发式偏低。

**全数据集权威结果**（400 GSM8K 样例，非 15% 分割）：
```
启发式 (τ=0.38): F1 = 0.8998
错误样例检测:  179/207 = 86.5%
正确样例保留:  181/193 = 93.8%
```

> 两个域均使用**启发式阈值 τ=0.38–0.44**，无需 RL 控制器。
> 离线小分割 RL 信号不足；在线大规模 RL 仍有潜力，但需要不同的训练设置。

- 需要更强 PRM 评分器（PBR32 或更好）

### 12.2 BCR/ABR 思想的重解释

**从"价值平滑性 = 步骤正确性"到"控制器路由"的转变**：

| 原始假设（Phase C）| 当前证据 | 修正结论 |
|------------------|---------|---------|
| Bellman 一致性可近似步骤正确性 | AUC/F1 表明需要真实步骤标签 | 不适合作为主训练目标 |
| 插值奖励可替代 PRM 监督 | 高质量标注数据是不可绕过的瓶颈 | 标签质量优先 |
| token-level BCR 足够 | 步骤级 first-bad-edge 更有效 | 步骤级监督是必要条件 |

**仍然有价值的 ABR 部分**：
- "何时验证"是独立的决策问题 → Phase F F2：ABR-lite 控制器（F1=0.863，PASS）
- 启发式阈值 τ=0.38–0.44 是目前最优的 ABR 实现
- BCR 差分形式（`|V(t) - V(t+1)| > δ`）vs ABR 绝对形式（`V(t) < τ`）是值得研究的控制器变体
- 在线 RL 框架下（PRIME 风格）BCR/ABR 仍可作为 routing gate
