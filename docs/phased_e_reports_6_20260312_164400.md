## 1. 项目主问题在 D/E/F 阶段的重新定义

### 1.1 初始朴素版本

较早期的隐含问题可以概括成：

1. 能否训练一个 `value head / process scorer`，
2. 让它对 reasoning prefix 的“好坏”排序，
3. 并最终用于 `ABR/BCR` 或更强的 RL 控制。

### 1.2 D/E/F 之后更准确的问题定义

到当前为止，真正的问题已经收缩成更具体的版本：

1. 能否学到一个**低成本但可信边界明确**的 process verifier / student scorer；
2. 它是否能从 `same-source learnability` 迁移到：
   - `ProcessBench`
   - `PRMBench`
   - `Phase F` controller / reward use；
3. 在不放弃 `ABR` 核心思想的前提下，
   - 如何把它从“单一 scalar head”升级为一个更完整的 verifier system。

### 1.3 当前主判断

截至本报告写作时：

1. `ABR` 核心思想没有过时。
2. 过时的是把它近似成：
   - `frozen backbone + one scalar head + pair ranking = 全部 verifier`
3. 当前主线更应理解为：
   - `ABR` 作为 cheap student / router / conservative scorer，
   - 外层再加：
     - answer verifier
     - uncertainty / abstain / escalate
     - stronger teacher / disagreement mining
     - process-outcome alignment gate

## 2. Phase D：方法学纠偏与 bridge 证据阶段

### 2.1 D 阶段为什么存在

`Phase D` 的核心不是“直接靠 teacher 把问题做完”，而是：

1. 用高质量外部 PRM / pair 信号验证我们的 value learning 是否真的可学；
2. 判断 teacher / external pair 到底是：
   - 真正有用的桥梁，
   - 还是只会引入新的错误监督。

对应主文档：

1. `docs/phase_D_plan.md`
2. `docs/phase_D4ABC_external_pairs_tutorial.md`

### 2.2 D 阶段早期纠偏：teacher 路线不能直接升主线

`Phase D` 一个非常关键的中期诊断是：

1. 旧的 `q_teacher / q_fused` 路线不能被直接当主线；
2. 因为若 teacher 与 corruption/pair 对齐本身有覆盖缺口，
3. 那么“teacher + pair consensus 有提升”的叙述就会被污染。

关键风险包括：

1. `teacher_pair_available_ratio = 0.0` 类覆盖问题；
2. teacher domain mismatch；
3. 把 calibration / scalar regression 误当作 ranking 结构学习。

### 2.3 D6T：外部 pair learnability 被证明是真实的

`DT2_MATH_SHEPHERD_SEED3_STABLE` 是 D 阶段的核心正结果之一。

主 artifact：

| mean_ext_auc | 0.7857 |
| std_ext_pair_acc | 0.0075 |

解释：

1. 这说明 **external pair ranking 是可学的**；
2. 而且不是单 seed 偶然；
3. 这一步把“value head 根本学不会”这个最强负解释排除了。

### 2.4 D6T transfer fix：same-family 学得动，不代表回到 StrategyQA 就会好

### 2.6 Phase D 最终产出

`Phase D` 的真正产出不是一个可直接用于部署的模型，而是三条方法学结论：

1. external pair / same-family ranking learning 是真实可学的；
2. ranking-only 目标比 joint 标量目标更贴近我们真正要的能力；
3. `StrategyQA` 不能再当主监督主战场，只能当 bridge / OOD / canary。

这三条结论共同把项目推进到了 `Phase E`。

## 3. Phase E 第一波：同源 learnability 与 source 语义诊断

### 3.1 E 阶段最初要回答什么

`Phase E` 最开始不是追 RL-ready，而是先回答三个更基础的问题：

1. 哪些数学 process 数据源是真的可学？
2. 训练失败时，到底是 source 差、监督差、还是实现差？
3. 同源 learnability 是否能稳定复现？

主文档：

1. `docs/phase_E_plan.md`
2. `docs/phase_E_multisource_math_plan.md`

### 3.2 Math-Shepherd：同源 learnability 很强

`Math-Shepherd` 是最早、最稳定证明同源 learnability 的源。

代表性结果：

1. `E41 / E68` 系列在同源 held-out 上可以稳定达到高 pair acc；
2. 一条明确结果是：
   - `E68_MS_ACC95_JOINT_LOGIT_SEED42`
   - held-out `pair_acc = 0.972450`

对应 artifact：

1. `assets/artifacts/phase_e_logs/phase_e_ms_acc95_push_0310_2146/final_summary.md`

这说明：

1. trainer 没坏；
2. 数据也不是完全没有信号；
3. `Math-Shepherd` 适合做 same-family anchor。

### 3.3 PRMBench_Preview：同样可学，但任务语义不同

早期 source 对比已经显示：

1. `Math-Shepherd` held-out 很高；
2. `PRMBench_Preview` 同样可以训出很高 held-out；
3. 但它们对 benchmark transfer 的帮助方式不同。

用户前期诊断里记录的代表结果：

| source | held-out pair_acc | held-out auc |
|---|---:|---:|
| Math-Shepherd | 0.9610 | 0.8908 |
| PRMBench_Preview | 0.9483 | 0.8333 |
| R-PRM | 0.4379 | 0.4666 |

这一步的重要意义在于：

1. **不是所有高 acc source 都代表同一种 verifier 能力；**
2. source 语义差异会直接影响 benchmark-facing transfer。

### 3.4 R-PRM：一个非常典型的“不是 source 不行，而是底层契约伤了 source”的例子

`R-PRM` 是 E 阶段最典型的实现/契约诊断正例。

最初现象：

1. `Math-Shepherd`、`PRMBench_Preview` 都能训出高 acc；
2. `R-PRM` 却接近随机。

后续诊断发现：

1. `R-PRM` 平均长度约 `1425.6` tokens；
2. `94.1%` pair 超过 `1024`；
3. `11.7%` pair 在截断后 `chosen/rejected` 直接变成一样；
4. `1536/2048` 后该问题显著减弱甚至消失。

由此得到的结论不是“R-PRM 没信号”，而是：

1. 旧的 `direct_pair_legacy + max_length=1024` 合同不适配；
2. 同一套训练栈并不自动公平地对待长文本 source；
3. contract / truncation diagnostics 必须成为正式基础设施。

### 3.5 多源混训的早期结论

早期多源尝试并没有直接带来强 benchmark 结果，原因包括：

1. source 语义不统一；
2. 不同 source 的 pair 几何并不相同；
3. 统一 objective 可能把 local / terminal / long-form verifier style 混在一起。

因此 E 阶段很快从“多源一起堆”转向了“先查语义与 benchmark 对齐”。

## 4. Phase E 第二波：ProcessBench 迁移困难的系统诊断

### 4.1 为什么需要重新审视 ProcessBench

早期一个危险的误判是：

1. same-source 指标高，
2. 就以为 verifier 已经接近可用。

但 `ProcessBench` 与训练契约并不对齐。

主要不对齐点：

1. 大量 `all-correct` 样本；
2. 不只是 `lastsafe_vs_firstbad`；
3. 还包含：
   - good prefix vs later bad
   - final correct but derivation flawed
   - critic-style error pinpointing

对应主诊断文档：

1. `docs/processbench_transfer_20260311.md`
2. `docs/processbench_state_and_community_gap_20260311.md`
3. `docs/phase_e_literature_refresh_20260311.md`

### 4.2 E79-E87 repair wave：局部修补都有效，但没有形成新主线

这一波实验主要是回答：

1. fanout 是否有效；
2. grid 是否有效；
3. terminal anchor 是否必要；
4. PRMBench mix 会不会补 benchmark。

结论不是“全部失败”，而是：

1. `fanout` 能修部分 `first_edge`；
2. `terminal anchor` 能显著抬 `terminal_top1`；
3. 但它们都很容易伤害 broader ranking；
4. `PRMBench mix` 对 `ProcessBench Math` 甚至可能是负迁移。

因此这一波的定位应写成：

1. **failure-mode discovery**
2. 不是 **mainline solution**

### 4.3 C3 / C4 / curated 线：混合监督比单轴修补更有希望

后来进入 curated/mixed 方向：

1. `C3 curated gated`
2. `C4 dual_head`
3. 以及更系统的 `mixed supervision`

它们证明了两点：

1. local / terminal 是可分解子任务；
2. 但简单 dual-head / hard routing 仍然会牺牲 broader ranking。

`C4 dual_head` 的意义主要不是“它赢了”，而是：

1. 证明了 `terminal` 与 `local` 的确需要拆；
2. 但当前这版架构还不够。

### 4.4 一个关键负结果：官方风格 F1 与内部 AUC 不能混读

E 阶段后来补做了一件非常重要的事：

1. 用现有 `scored_rows.jsonl` 回算更接近官方的 `ProcessBench F1`；
2. 结果发现很多“看起来不错”的 run 实际 official-style F1 并不高。

这一步把以下反模式正式钉住了：

1. 只报 `pair_acc`；
2. 只报 `AUC`；
3. 只报 oracle-threshold F1；
4. 把这些粗暴类比为“已经 benchmark-ready”。

截至 2026-03-12，可确认的近期 verifier frontier 线主要是：

1. `PBR26`

   - `ProcessBench F1 (Math)` 已到 `0.686` 量级。
2. `PBR31`
   - 作为 stronger slice / controller research 候选被反复使用。
3. `PBR32`
   - all-layer LoRA + Math-PRM backbone 路线的重要候选；
   - 但截至当前，Math 上未证明显著超过 `PBR26/PBR31`。
4. `PBR33`
   - `top-4 LoRA + PBR26 data`
   - 当前已知结果：`Math F1=0.666`, `GSM F1=0.797`
   - 说明 LoRA 能在 GSM 上给出一些增益，但未突破 Math ceiling。

### 5.6 当前对 LoRA 的判断

截至本报告写作：

1. `LoRA` 仍值得继续；
2. 但“LoRA 一开就自然突破 frozen ceiling”并没有被证明；
3. 当前更合理的判断是：
   - 数据几何仍然是第一驱动；
   - backbone adaptation 是必要但不充分；
   - LoRA 很可能会放大训练 pair 分布偏置。

## 6. Phase E 实现与基础设施审计

### 6.1 已确认并修复的高危问题

这一部分对论文“实验可信度”非常重要。

已确认并修掉/加闸门的高危问题包括：

1. `feature cache` 活锁/抢锁问题
   - 活着的写进程会被 stale-lock 逻辑误抢。
2. `best -> final` 静默 checkpoint fallback
   - Phase B/E 都涉及过。
3. `Phase A official split` 泄漏
   - 准备数据时可能按请求 split 而不是 loader 真实 split 写出。
4. `ProcessBench F1` 口径过度依赖 oracle threshold
   - 后来已补 provenance 记录与 fixed-threshold 对照。
5. `R-PRM` 长文本截断合同问题
   - 旧 `1024` 合同直接系统性伤害该 source。

### 6.2 审计后的整体判断

当前仓库已经不适合继续用“实现到处都有 hidden bug”作为默认解释。

更准确的说法是：

1. 大规模语法级/跑不起来问题已不是主因；
2. 真正危险的是：
   - 回退路径是否被显式记录；
   - 数据语义是否与 benchmark 对齐；
   - 评测阈值是否诚实；
   - 长度、缓存、pair 契约是否污染结果。

## 7. Phase F：从 verifier 到 controller / RL 的现实检验

### 7.1 Phase F 一开始想回答什么

`Phase F` 关注的问题不是“ProcessBench 再涨几分”，而是：

1. 当前 verifier 是否已经能支持可用的 `ABR-lite / BCR-lite controller`；
2. 是否值得继续往 RL controller 或 live RL 推进；
3. 哪些失败来自 verifier 不够强，哪些失败来自 controller / reward 设计错误。

### 7.2 F1 preflight：阈值稳定性与 reward-hacking probe 是必要前置

当前 F 阶段最重要的一个共识是：

1. 在 live RL 或 live controller 之前，必须先做：
   - threshold / shift audit
   - reward hacking probe
2. 否则容易把“某个 split 上阈值刚好凑巧有效”误当可部署。

### 7.3 旧 controller 结论一度过于悲观，但后来被修正

早期一个负结果是：

1. 某些 aggressive baseline controller 会过度拒绝 all-correct 样本；
2. 因而看起来像“verifier 不适合 controller”。

但后来的更系统 sweep 发现：

1. 问题不全在 verifier；
2. 很大一部分在 controller family 本身设计得太差。

### 7.4 offline controller sweep：简单 heuristic 明显优于旧 baseline

关键 artifact：

1. `assets/artifacts/phase_f_controller_sweep/phase_f_controller_sweep_0312_main_20260311T181216Z/summary.md`

代表性结果：

| case | old baseline | best redesigned policy | balanced_f1 |
|---|---:|---|---:|
| pbr19_math | 0.5537 | threshold_only tau=0.38 | 0.8674 |
| pbr21_math | 0.5588 | threshold_only tau=0.35 | 0.8641 |
| pbr26_math | 0.5752 | threshold_only tau=0.38 | 0.8639 |
| pbr19_gsm | 0.6713 | delayed_drop | 0.9052 |
| pbr21_gsm | 0.6405 | guarded_drop | 0.9028 |
| pbr26_gsm | 0.6509 | delayed_drop | 0.9053 |

结论：

1. `Phase F` 不是“verifier 太弱所以 controller 必败”；
2. 相反，**简单 heuristic controller 已经很强**；
3. 这也说明：
   - controller RL 不能再默认比 heuristic 更高级。

### 7.5 BC / BC->RL / from-scratch RL-like：teacher structure 仍是关键

来自 `2026-03-12` 的更新结果：

1. `bc_only`
   - `p31_math = 0.7987`
   - `p32_math = 0.8309`
   - `p31_gsm = 0.9000`
2. `bc_then_rl`
   - `p31_math = 0.8171`
   - `p32_math = 0.8395`
   - `p31_gsm = 0.9082`
3. from-scratch RL-like
   - `p31_math = 0.0000`
   - `p31_gsm = 0.0000`

主结论：

1. teacher structure 是必要的；
2. `BC` 已经很强；
3. 温和 `BC -> RL` 可以在一些 held-out split 上带来小增益；
4. from-scratch RL 仍然极不稳定，不该升主线。

### 7.6 cheap-to-strong gating：只在部分 slice 上有意义

`2026-03-12` 的 cheap-to-strong gating 结果表明：

1. 在 Math 上，当前“更强 verifier 候选”并不真的更强；
2. 因而 cheap-to-strong gating 在 Math 上暂时没有收益；
3. 在 GSM 上，有限比例 escalation 有小幅收益，且预算可控。

这一步非常重要，因为它说明：

1. `cheap -> strong` 是正确系统设计方向；
2. 但它依赖于“strong truly stronger”；
3. 目前在 Math slice 上，这个条件还没满足。

### 7.7 GRPO feasibility：信号并不是零，但 RL 仍不能盲目推进

`2026-03-12` 的 mixed-pool feasibility 修正后结果显示：

1. Math (`PBR32` rows):
   - process `|adv| = 0.811`
   - outcome `|adv| = 0.864`
2. GSM (`PBR33` rows):
   - process `|adv| = 0.805`
   - outcome `|adv| = 0.872`

因此：

1. 当前 RL blocker 不是“reward 根本没有 variance”；
2. 更可能是：
   - benchmark saturation
   - optimization stability
   - reward shaping / hacking
   - verifier quality on hard slices

### 7.8 当前 Phase F 的真实结论

当前最准确的 Phase F 结论应写成：

1. `heuristic controller` 已经可用，而且比以前想象得更强；
2. `BC / BC->RL` 值得保留，属于 practical distillation 方向；
3. from-scratch RL controller 暂不值得押注；
4. live RL 仍然需要：
   - 更好的 verifier
   - 更诚实的 gate
   - 更安全的 reward shaping

### 8.1 被排除掉的错误解释

1. “代码实现大面积坏掉所以结果不可信”

4. “同源高 acc 就足够说明 RL-ready”

当前最可信的解释链是：

1. 训练栈能学；
2. same-source / same-family learnability 为真；
3. benchmark / RL-facing 难点主要来自：
   - 数据几何不对齐
   - source 语义混杂
   - terminal / all-correct / equivalence 未显式拆出
   - verifier 角色设计过窄
   - controller / RL 设计曾过度复杂化

### 8.3 当前最关键的反模式

1. 用 same-family 高 acc 代替 benchmark transfer。
2. 用 oracle-sweep F1 代替 fixed-threshold deployment metric。
3. 在 mixed semantics 数据上继续堆 head 复杂度，而不先修数据几何。
4. 让单一 cheap scorer 在本该 abstain 的 slice 上强行给分。
5. 把 from-scratch RL 误当默认更先进路线。

### 12.2 2026-03-12 17:10 live literature refresh and next-10h queue

这轮又补做了一次面向 `2025H2 -> 2026Q1` 的联网刷新，重点不是重复论证“PRM 有价值”，而是确认社区对我们当前这些顽固问题是否已经形成更清晰的共识。

#### A. 新鲜外部证据与对本项目的直接含义

1. `When to Trust the Cheap Check: Weak and Strong Verification for Reasoning`
   - `https://arxiv.org/abs/2602.17633`
   - 关键启发不是“再把 weak verifier 训得更大”，而是：
     - cheap verifier 应作为第一层过滤，
     - 对不确定样本必须保留 `abstain / escalate / strong verification`
     - 真正重要的是 `calibration + confidence shaping`
   - 对本项目的含义：
     - `Phase F` 不应继续把单一 verifier score 直接等同于最终控制信号；
     - 当前 `ABR/BCR` student 更适合作为 cheap verifier。

2. `PRIME: A Process-Outcome Alignment Benchmark for Verifiable Reasoning in Mathematics and Engineering`
   - `https://arxiv.org/abs/2602.11570`
   - 直接证据是：
     - 过程-结果错配会系统性误导 outcome-only 奖励；
     - verifier 在 `process-outcome alignment` 上的能力，与后续 RLVR 效果强相关。
   - 对本项目的含义：
     - 后续 gate 不能只报 `ProcessBench AUC`；
     - 必须补 `process-outcome alignment` 风格切片。

3. `VerifyBench: A Systematic Benchmark for Evaluating Reasoning Verifiers Across Domains`
   - `https://arxiv.org/abs/2507.09884`
   - 关键发现是 verifier 对：
     - 输入格式
     - 长度
     - 域迁移
     - prompt template
     都很敏感。
   - 这和我们已知问题完全同向：
     - `R-PRM` truncation contract
     - `ProcessBench` local / terminal mismatch
     - `PBR32` slice rebalance

4. `GenPRM`
   - `https://arxiv.org/abs/2504.00891`
   - 强信号是：
     - 更高上限往往来自 generative verifier、
     - critique / code check、
     - test-time scaling，
     - 而不是单纯更复杂的 scalar head。
   - 对本项目的含义：
     - frozen feature scalar head 更适合作为 cheap student / routing score；
     - 若要进一步冲 benchmark 上限，需要考虑 generative verification 路线。

5. `ThinkPRM`
   - `https://arxiv.org/abs/2504.16828`
   - 关键价值是：
     - 少量高质量步骤监督
     - synthetic verification rationale
     - verifier 也能“think longer”
   - 对本项目的含义：
     - 数据 curate 不应只继续堆 pair 数；
     - 更应追求高信息量 verification rationale / disagreement samples。

6. `LongRLVR`
   - `https://openreview.net/forum?id=omVhYvyTPJ`
   - 关键结论是长上下文 RL 只靠 final-answer reward 太 sparse，需要过程或上下文奖励。
   - 对本项目的含义：
     - `Phase F` 若继续推进 RL，不应回退成纯 outcome reward。

7. `Hybrid Reinforcement: when reward is sparse, better to be dense`
   - `https://openreview.net/forum?id=0CajQNVKyB`
   - 方向上强调：
     - verifier reward 与 dense / model-based reward 的混合，
     - 常优于单一来源。
   - 对本项目的含义：
     - `ABR` 核心可以保留，
     - 但要允许与 terminal / answer-level / strong verifier signal 混合。

8. `RLPIR`
   - `https://openreview.net/forum?id=3jrWUqoItt`
   - 提醒我们 prefix rollout / intrinsic consistency 也可作为 verifier 稀缺时的补偿信号。
   - 对本项目的含义：
     - 真正到在线 RL 阶段时，prefix rollout 仍值得保留为备选。

#### B. 这轮刷新后对研究方向的收束

这轮没有出现证据支持“继续 head-only sweep 就能解决 stubborn transfer”。相反，社区信号更一致地指向：

1. `Phase E` 当前 head 更适合当：
   - cheap verifier

   - promotion gate 是否诚实
   - 是否存在 abstain / escalate 机制
   - 是否覆盖 process-outcome alignment
