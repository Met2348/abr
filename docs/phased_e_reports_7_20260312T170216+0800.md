## 1. Executive Summary

This report consolidates the major technical attempts from the later `Phase D` bridge period through the current `Phase E/F` verifier-system redesign.

The most important stable conclusions so far are:

1. `Phase D` proved that external ranking supervision can help `StrategyQA`, but only under a ranking-only bridge. Joint scalar-calibration style objectives regressed the target metric.
2. `Phase E` proved that same-source value/ranking learning is real on high-quality pair sources, but benchmark-facing trustworthiness is much harder than same-source held-out fit.
3. `Math-Shepherd` and `PRMBench_Preview` are currently the strongest same-source sources. `R-PRM compact_verdict` remains learnable on train distribution but weak on true held-out, so it is not a trustworthy main anchor under the current contract.
4. Several implementation-level silent risks that could invalidate conclusions were found and fixed:
   - feature-cache provenance too loose,
   - stale cache and stale lock behavior,
   - whole-cache-on-GPU behavior in training,
   - silent `best -> final` checkpoint fallback,
   - dangerous recipe defaults in `Phase E` trainers/wrappers,
   - summary-layer bugs in some `Phase E` and `Phase F` wrappers.
5. The current strategic bottleneck is no longer “can the model learn local ranking at all”. The bottleneck is:
   - terminal completion valuation,
   - process-outcome alignment,
   - benchmark-facing robustness,
   - whether the resulting verifier is useful enough to support later RL / controller stages.
6. The project has therefore already moved away from treating a single scalar frozen-head verifier as the final object. The emerging system direction is:
   - local/process verifier,
   - terminal/answer verifier,
   - abstain/escalate gate,
   - optional strong verifier / judge / deterministic verifier on hard slices.

## 2. Chronology And Phase Roles

### 2.1 Phase D

Primary role:
1. correct the methodology after early `Phase C` value-head failures,
2. test external pair supervision,
3. test whether external ranking knowledge can bridge back into `StrategyQA`.

### 2.2 Phase E

Primary role:
1. stop relying on weak `StrategyQA` pseudo-process supervision as the main training source,
2. validate same-source and same-family learnability on higher-quality math/process pair sources,
3. build a verifier stack that can eventually become `RL-ready`.

### 2.3 Phase F

Primary role:
1. turn `Phase E` verifier outputs into controller/reward infrastructure,
2. audit whether verifier quality is actually useful for thresholding, rejection, routing, and offline RL-like objectives,
3. reject verifier candidates that look good on same-source metrics but fail controller-facing or reward-hacking diagnostics.

## 6. Implementation And Infrastructure Audits

This project repeatedly found that result interpretation could be silently corrupted by infrastructure bugs. The following classes of issue were audited and fixed.

### 6.1 Feature Cache And Provenance Risks

Files:
- `src/ours/phase_b/feature_cache.py`
- `src/ours/phase_e/runtime.py`
- `scripts/phase_b_train_value.py`
- `scripts/phase_c_train_pik.py`

Fixes that already landed:
1. cache schema upgraded and provenance tightened;
2. actual weight shards are now part of signatures;
3. stale lock and corrupt cache recovery were added;
4. `Phase B/C/E` moved from whole-cache-on-GPU behavior to CPU-cache plus batch-to-GPU behavior.

Why this mattered:
1. old provenance could silently reuse wrong features after model changes;
2. whole-cache GPU residency could create misleading OOM behavior and unstable training.

### 6.2 Checkpoint Resolution Risk

Files:
- `scripts/phase_b_eval_faithfulness.py`
- `scripts/phase_c_eval_pik.py`
- `scripts/phase_d_eval_external_pairs.py`
- `src/ours/phase_e/runtime.py`

Fixes that already landed:
1. `best -> final` fallback is no longer silent;
2. default missing policy is strict `fail`;
3. any fallback must be explicit and is written into metrics / manifest.

Why this mattered:
1. silent fallback could make two experiments appear to compare `best` checkpoints while actually evaluating different endpoints.

### 6.3 Recipe Safety And Collapse Diagnosis

Files:
- `src/ours/phase_e/recipe_safety.py`
- `scripts/phase_e_train_value.py`
- `scripts/phase_e_diagnose_training_collapse.py`
- many `run_phase_e*.sh` wrappers

Fixes:
1. known catastrophic recipe combinations are now blocked by default;
2. wrappers explicitly pass `--recipe-risk-policy`;
3. training emits structured health diagnostics.

Why this mattered:
1. bad recipes had previously polluted source-quality comparisons.

### 6.4 Semantic Contract Risk In Math-Shepherd Step Pairs

Key audit finding:
- the mentor correctly identified that some earlier pair construction logic was using cross-depth prefixes from one trajectory rather than true local same-step supervision.

Fix:
- step-label pair construction was moved to `first_bad_edge_strict` semantics;
- legacy nearest-negative semantics were downgraded to lower-trust historical evidence.

Artifacts and docs:
- `docs/data_semantics_risk_audit_20260310.md`
- corresponding `phase_e_pairs_v2` / `phase_d_external_pairs_v2` artifacts.

## 9. Current Project Bottlenecks

The most stubborn problems are now:

1. `terminal blind spot`
   - strong local ranking does not imply proper valuation of all-correct terminal completion.
2. `process-outcome alignment`
   - a verifier can fit source pairs while still failing benchmark-facing process judgment.
3. `cheap verifier insufficiency`
   - current weak verifiers are not yet strong enough to make cheap-to-strong escalation cost-effective.
4. `source contract mismatch`
   - some sources are learnable but not trustworthy.
5. `representation ceiling`
   - still plausible in some lines, motivating continued controlled LoRA probes.

## 11. Practical Writing Guidance

When writing the paper or slides, the safest narrative is:

1. early scalar-value attempts on weakly aligned supervision were unstable or misleading;
2. methodology and infrastructure had to be corrected before source comparisons became trustworthy;
3. once corrected, same-source learnability on high-quality pair sources was clearly established;
4. however, RL-ready verifier trust requires more than same-source high accuracy;
5. therefore the project is evolving from “one scalar head” toward a verifier system.

A good concise statement is:

> The project has already established that high-quality pair/process supervision can train strong same-source ranking utilities, but current evidence still falls short of demonstrating a universally trustworthy RL verifier. The remaining bottleneck is process-outcome alignment and terminal completion valuation, not the basic existence of learnable local ranking signal.
