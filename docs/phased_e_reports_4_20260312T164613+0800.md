## 1. 总体结论先行

到当前时点，项目已经经历了三个本质不同的阶段：

1. `Phase D`：
   试图用外部 PRM / 外部 step data 把 value head 从“几乎随机”拉起来。
   核心收获是：
   - 单纯 outcome-style / weak-MC / teacher score 融合不够；
   - 真正有效的是更高质量的 pair geometry，尤其是 `sibling-branch` 类型监督。

2. `Phase E`：
   从“弱价值头”转向“benchmark-facing verifier system”设计。
   核心收获是：
   - `Math-Shepherd strict-only` 不够；
   - `R-PRM` 在当前 compact scalar-head 契约下并不是强源；
   - `Math-Step-DPO / sibling-branch` 是目前最重要的正信号；
   - `Qwen2.5-Math-PRM-7B` 作为冻结 backbone 是真正的突破；
   - `PRMBench local + bounded terminal auxiliary (+ small grid support)` 是当前最强的 benchmark-oriented 数据几何。

3. `Phase F`：
   从“这个 verifier 能不能做 RL 奖励”转向“它先能不能做 deployment-grade controller / reranker / pre-RL infrastructure”。
   核心收获是：
   - 旧 `ABR-lite` 控制规则确实不行；
   - 但 verifier 本身并不是主问题；
   - 换成更合理的 controller family 以后，offline controller 指标大幅恢复；
   - 当前最自然的下一步不是直接大规模 RL，而是：
     1. `live heuristic controller`
     2. `BC-distilled controller`
     3. 再考虑小规模 RL 微调

当前最新状态：
1. `Phase E` 最有价值的新线是 `Math-PRM backbone + ProcessBench hybrid (PH1/PH2/PH3 family)`。
2. `Phase F` 最有价值的新线是 `heuristic / BC controller usability`，而不是从零开始 policy-gradient。
3. 当前还不能严肃宣称“已进入可放心作为 RL 主奖励的阶段”；但可以说：
   - verifier 已经进入“值得认真做 RL 前置验证”的强候选区间。

## 2. Phase D：从外援 PRM 到高质量 pair 监督的转向

### 2.1 初始背景

进入 `Phase D` 时，仓库的核心问题是：
1. `Phase C` 训练出来的 value head 在 `pair_acc / AUC` 上基本接近随机；
2. 即使校准指标（如 Brier）有所改善，也无法转化成真正的过程判别能力；
3. 项目开始意识到：
   - 不是“模型完全不会学”，
   - 而是“标签和训练目标很可能不对”。

### 2.2 D4ABC：外部 pair / 外部 PRM 的首次系统引入

目标：
1. 引入外部过程监督资源，改善 Phase C 自造 pair 的低质量问题；
2. 通过 `A/B/C` 对照实验，确认哪些数据源真的提供有用信号。

代表性入口：
- `scripts/run_phase_d_external_pair_suite.sh`
- `src/ours/phase_d/external_pairs_adapters.py`

核心方向：
1. `A`: 直接 external pair 资源
2. `B`: step-label 数据转 pair
3. `C`: 多源混合 / 权重组合

### 2.3 早期结论：不是所有“外援”都一样有用

这一步最重要的结论不是“外部数据一定有用”，而是：
1. `R-PRM` / `PRMBench_Preview` 这类显式 pair 资源更像真正的偏好数据；
2. `Math-Shepherd` 这种 step label 数据，如果转换方式不对，会把“prefix 深度差异”误当成“步骤优劣差异”；
3. `teacher score` 作为总分，不足以直接解决排序问题。

### 2.4 D6T：最小可行验证（triplet / ranking-only）

代表方向：
- `run_phase_d_triplet_validation_suite.sh`
- `DT1 / DT2 / DT3 / DT4 / DT6` 等组

最关键的历史突破之一：
- `DT1 Math-Shepherd smoke` 曾经拿到：
  - `pair_accuracy ≈ 0.73`
  - `auc ≈ 0.80`

它回答的问题不是“项目已经成了”，而是：
1. 高质量 triplet / ranking 训练是能学起来的；
2. 旧的“完全随机”并不是 architecture 天生无解；
3. 问题更多出在：
   - pair 构造
   - 训练目标
   - 数据契约

### 2.5 D 阶段最重要的失败诊断

D 阶段最关键的几条诊断后来被反复验证：

1. `Math-Shepherd` 早期转换方式有重大语义风险。
   - 旧逻辑本质上在比较不同长度 prefix；
   - 这会混入“离终点更近”而不是真正“步骤更对”的信号。

2. `R-PRM` 早期表现差，不只是源数据差，還叠加了：
   - 长输入被 `1024` 截断；
   - compact scalar-head 契约不匹配。

3. 只做 `teacher score` 融合，不会自动变成高质量排序监督。

### 2.6 D 阶段给 E 阶段留下的真正资产

D 阶段最重要的遗产不是某个单一 best checkpoint，而是这三条路线判断：

1. 以后必须更重视 `pair geometry`。
2. 必须区分：
   - local first-bad discrimination
   - broader prefix ranking
   - terminal completion ordering
3. benchmark-facing verifier 不能再只靠单一 `strict-only` 或 `teacher-score-only` 路径。

## 3. Phase E：从“训练一个小 value head”到“设计 benchmark-facing verifier system”

### 3.1 E 早期：same-source 变强，但 benchmark 迁移差

最早一批 `Phase E` 单源实验已经表明：
1. 同源 held-out 可以很高；
2. 但一到 `ProcessBench` 就接近随机或显著下降。

这说明：
1. 旧 pipeline 并不只是“模型训练失败”；
2. 更大的问题是 `训练监督几何和 benchmark 目标不对齐`。

### 3.2 Math-Shepherd 修复：修掉主 bug，但没有解决全局问题

后续修正把 `Math-Shepherd` 从“不同深度 prefix 乱配对”改成更合理的 `first_bad_edge_strict`。

这一步的重要结论是：
1. 旧 bug 的确真实存在，必须修；
2. 修掉以后，same-source 结果改善，但 `ProcessBench` 仍未根本解决；
3. 所以主问题并不只是一个实现 bug，而是更深层的方法错位。

### 3.3 intradataset ACC90：source 之间的可学性差异

`I2 / I3 / I4` 系列回答了一个很重要的问题：

1. `Math-Shepherd` same-source 是可学的；
2. `PRMBench_Preview` same-source 也是可学的；
3. `R-PRM` 在当前合同下明显更难。

这一步的重要收获：
1. 不同 source 的 supervision semantics 差异极大；
2. “外部 pair 数据”不能笼统地当成同一种东西；
3. `R-PRM` 需要单独诊断，不能和 `Math-Shepherd / PRMBench_Preview` 混为一谈。

### 3.4 R-PRM 诊断：从截断灾难，到监督契约不匹配

早期 `R-PRM` 的弱结果有两个阶段：

#### 阶段 1：长度截断问题

在旧合同下：
1. 大量 pair 超过 `1024` token；
2. chosen / rejected 区分信息经常落在 cutoff 后；
3. 这是明确的输入污染问题。

#### 阶段 2：截断修正后仍然不强

改成 `compact_verdict` / `compact_correctness` 后，长度问题显著缓解；
但结果表明：
1. `R-PRM` 仍然只中等可学；
2. polarity bias 可以单独修掉；
3. 但修掉 bias 后，总体指标仍停在中等水平。

结论：
1. `R-PRM` 当前不是 Phase E 主训练锚点；
2. 更适合作为辅助诊断源，而不是核心 benchmark-facing mainline。

### 3.5 ProcessBench 对齐诊断：当前主问题被拆清楚

这是 `Phase E` 最关键的一步之一。

通过：
- `phase_e_analyze_processbench_failures.py`
- terminal anchor pilots
- all_good_vs_all_bad grid
- benchmark bucket analysis

最终定位出：

1. `ProcessBench` 不只是 local first-bad benchmark；
2. 它有大量 `all-correct` / final-completion 相关 slice；
3. 因此：
   - 纯 local 监督不够；
   - 纯 terminal 监督会过修；
   - 纯 broader grid 也不够。

这一步把 E 阶段问题正式收缩为：

> 如何同时兼顾：
> 1. local first-error detection
> 2. broader prefix ranking
> 3. terminal completion ordering

### 3.6 NDS 系列：新数据范式探索

这是 E 阶段的第二次大转向。

#### NDS5：Math-Shepherd strict-only

结果：
- benchmark 迁移很差

结论：
1. 不是简单去掉 grid/fanout 就能解决问题；
2. strict-only 几何太窄。

#### NDS6：RLHFlow strict-only

结果：
- 比 `NDS5` 略好，但仍不足

结论：
1. 更好的标签有帮助；
2. 但 `strict-only` 仍不是对的几何。

#### NDS7：Math-Step-DPO / sibling-branch

这是一个真正的范式突破。

结果表明：
1. `sibling-branch` 对 `ProcessBench` 更对齐；
2. `first-edge` 和 `good-vs-bad` 都明显改善；
3. `gated_mlp` 在这条监督上确实能进一步提升。

从这里开始，项目的主叙事发生了改变：
1. 不是“我们还缺更好的 head”；
2. 而是“我们终于找到更对的监督几何”。

### 3.7 Math-PRM-7B backbone：真正的 backbone breakthrough

这是 E 阶段迄今最重要的突破。

代表性结论：
1. `Qwen2.5-7B-Instruct` -> `Math-7B-Instruct` -> `Qwen2.5-Math-PRM-7B`，
   backbone 表示能力是分档提升的；
2. 用 `Math-PRM-7B` 作为冻结 backbone，比多数 head 架构微调的提升更重要；
3. 这和社区对“专化 PRM backbone hidden states 更适合步骤验证”的认识高度一致。

关键影响：
1. 以后不再把普通 `7B-Instruct` 视为主线；
2. `Math-PRM-7B` 成为 E 阶段 benchmark-facing verifier 的新主 backbone。

### 3.8 PBR / PRX 系列：三角竞争格局

在 `Math-PRM-7B` backbone 下，出现了三类强候选：

1. `PBR12`
   - `Math AUC` 强
2. `PBR21`
   - oracle F1 / overall balance 很强
3. `PRX1`
   - fixed-threshold GSM 很强

这说明：
1. 当前不是单一 SOTA 一锤定音；
2. 而是一个“不同 deployment metric 下的三角格局”。

同时，也给 Phase F 一个重要提醒：
1. 进入 RL / controller 之前，必须明确到底优化的是哪个 deployment criterion；
2. 否则会被 oracle 指标误导。

### 3.9 PH1 / PH2：最新 PRM-backbone hybrid 路线

这是当前最前沿、也是最值得继续投资源的 E 阶段路线。

#### PH1

结构：
1. `PRMBench local` 作为主 anchor
2. `terminal auxiliary (15%)`
3. 结构对比：`mlp` vs `gated_mlp`

最新结果（128-sample smoke，但已经很强）：
- `pb_gsm_auc ≈ 0.855 / 0.826`
- `pb_math_auc ≈ 0.855 / 0.849`
- `pb_math_all_correct ≈ 0.775 / 0.814`

解释：
1. `PH1` 在 Math 侧特别强；
2. terminal 覆盖更好；
3. 对 all-correct slice 更友好。

#### PH2

结构：
1. `PRMBench local`
2. 更轻的 `terminal auxiliary (10%)`
3. 额外加 `MS-grid 10%`

最新结果：
- `pb_gsm_auc ≈ 0.861 / 0.843`
- `pb_math_auc ≈ 0.849 / 0.846`
- `pb_gsm_first_edge ≈ 0.943`

解释：
1. `PH2` 在 GSM 上非常强；
2. broader prefix coverage 带来 GSM pair/AUC 收益；
3. 但 terminal side 比 `PH1` 略弱。

#### PH1 vs PH2 当前结论

1. `PH1` 更像 Math / terminal-friendly 路线；
2. `PH2` 更像 GSM / broader-prefix-friendly 路线；
3. 因此最合理的下一步是：
   - 做桥接混合（如 `PH3`）
   - 再把这批 hybrid verifier 用于 F 阶段 controller/usability 验证。

## 4. Phase F：从“RL-ready?”到“先把 usable controller 做出来”

### 4.1 最早的误区：把 verifier 问题直接跳到 RL 问题

最开始的直觉是：
1. 如果 verifier 够强，直接给 RL 用；
2. 但现实中很快发现：
   - reward hacking
   - threshold brittleness
   - generator shift
   - terminal slice fragility
   这些 deployment 问题远早于真正的 online RL 收益。

### 4.2 modern preflight：真正可信的 RL 前置门

`Phase F modern preflight` 的意义是：
在进入 RL 之前，先审查：
1. threshold / shift
2. reward hacking surface

最新可信 summary 表明：
1. `PBR26 / PBR31` 在固定阈值附近有一定可用性；
2. 但仍有明显 generator-shift 风险；
3. `confidence_tail / filler_tail` 等攻击在部分 slice 上能抬高 first-bad prefix 分数。

解释：
1. “能排序”不等于“适合作 RL reward”；
2. preflight 是必要门槛，不是可选项。

### 4.3 ABR-lite 旧规则失败，但 verifier 不是主问题

旧的 `baseline_immediate` / 老式 `ABR-lite` controller：
1. 过于激进地 early-stop；
2. 严重误杀 all-correct 轨迹；
3. 导致最初得出“F2 不行”的结论。

但后续 controller sweep 改写了这个结论：
1. 不是 verifier 不行；
2. 而是控制规则不对。

### 4.4 controller family sweep：Phase F 的第二次方法转向

这一步是 F 阶段的关键突破。

重要结论：
1. `threshold_only`
2. `delayed_drop`
3. `guarded_drop`

这些简单规则已经能把 offline controller 指标拉回高位。

典型结果：
- `Math`: `threshold_only` 很强
- `GSM`: `delayed_drop / guarded_drop` 更强

这一步把问题重新定义为：
1. 不再是“Phase F 不可用”；
2. 而是“应该先做 heuristic controller / controller family selection”。

### 4.5 generator robustness / ensemble：F 阶段进一步收口

后续又验证了：
1. worst-generator robust policy 是有价值的；
2. weak-verifier ensemble 还能进一步推高稳健性。

这和社区近两年的 `abstention / selective control / cheap->strong verification` 方向高度一致。

### 4.6 trainable controller：RL-like 方向的真正边界

后续探索包括：
1. `from-scratch REINFORCE`
2. `behavior cloning`
3. `BC -> RL fine-tune`
4. robust-from-scratch RL-like

最清楚的结论是：
1. 从零开始 RL-like controller 非常容易 collapse；
2. `BC-only` 已经很强；
3. `BC -> RL` 目前通常不如 `BC-only` 稳定；
4. 所以 controller 这条线的合理排序是：
   - heuristic
   - BC
   - BC + tiny RL
   - 纯 RL from scratch（当前不推荐）

### 4.7 GRPO / RL feasibility：当前不能过早宣传

当前 F 阶段还有一些历史尝试：
1. outcome-only GRPO
2. process reward clip-delta
3. BoN rerank
4. LoRA / frozen scorer 的 generation-time discriminability

这些实验给出的现实结论是：
1. 高 baseline accuracy 的 benchmark（如 GSM8K 高位）不适合继续做 GRPO 提升验证；
2. 过程奖励如果用于 RL，必须非常谨慎地做 reward shaping；
3. 当前更适合说“值得认真做 RL 前置验证”，
   而不是说“已经 ready for RL main reward”。

## 5. 代码与基础设施层面的关键修复

虽然当前主问题已偏研究设计，但仍有几类高危实现问题曾经真实存在并影响结论。

### 5.1 Math-Shepherd pair semantics bug

问题：
1. 旧 pair 构造把不同深度 prefix 当成 step-quality 对比；
2. 这会混入长度/进度信号。

影响：
- same-source 可学，但 benchmark 解释错位。

修复后结论：
- 修了必要，但不是根本解决方案。

### 5.2 R-PRM truncation contract

问题：
1. 长 verifier-style text 在 `1024` 截断下严重失真。

修复后结论：
1. 截断是早期主因之一；
2. 但修完后 `R-PRM` 仍不是强源。

### 5.3 compare / selector / launcher provenance bugs

曾经出现的几类问题：
1. compare 脚本假定 `gap5p` 总存在；
2. selector 参数展开不一致，导致候选选错；
3. overnight launcher 只等待 `final_summary.md` 文件存在，不看 `status: ok`。

这类问题的重要性在于：
1. 它们未必破坏训练本身；
2. 但会严重污染“第二天早上如何解释实验”的 provenance。

### 5.4 Phase F old metric reporting bug

问题：
1. 曾经把 `F2 ABR-lite` 过早写成 strong pass；
2. 实际 raw artifact 不支持这个结论。

修正后：
1. 正式结论改成：
   - 旧 controller 不行；
   - 新 controller family 离线很强；
   - live controller 仍需验证。

### 5.5 当前实现层总判断

到当前为止，A-F 主链的实现质量已经明显高于“随意拼接”的水平。

现在更大的风险是：
1. 研究解释错误；
2. 阈值/部署口径不诚实；
3. 数据语义和 benchmark 几何不对齐；
4. 把局部成功误当成 RL-ready。

## 6. 社区文献和当前项目最一致的结论

### 6.1 社区对当前问题的共识

结合 `ProcessBench / VerifyBench / AbstentionBench / PRIME / Rewarding Progress / Tree-PLV / Lessons of Developing PRMs / ThinkPRM / GenPRM` 等来源，可以把当前问题概括成：

1. `outcome-only` 很难支撑强步骤验证；
2. `strict-only` 标签通常不足以覆盖真实 benchmark；
3. `sibling-branch / preference` 数据更有价值；
4. verifier 需要区分：
   - local process errors
   - final completion correctness
   - answer equivalence / invalidity
   - abstain / escalate
5. 对 RL 而言：
   - reward hacking surface
   - threshold stability
   - policy shift robustness
   这些必须先被审计。

### 6.2 当前项目与社区最契合的地方

1. `sibling-branch` 转向是对的；
2. `Math-PRM backbone` 转向是对的；
3. `controller / abstention / usability` 先于大规模 RL，是对的；
4. `cheap verifier + 少量升级` 当前还不够，需要更强结构分解。

### 6.3 当前项目与社区仍有差距的地方

1. 还缺独立 `answer verifier / invalid detector`；
2. 还缺明确的 `process-outcome alignment` 子模块；
3. 还缺真正的 `generative verifier / critic-style` 路线；
4. 还缺 live deployment / controller 验证闭环。

## 7. 当前最值得写进论文的主叙事

如果现在写论文或中期报告，最合理的叙事主线是：

### 7.1 负面发现

1. naive outcome / MC / weak-teacher 融合并不能稳定得到好 verifier；
2. `Math-Shepherd strict-only` 不足以迁移到 `ProcessBench`；
3. 单纯追求 same-source held-out 高分会误导研究方向；
4. `R-PRM` 在当前 compact scalar-head 契约下不是强训练锚点。

### 7.2 关键正面发现

1. `sibling-branch` supervision 是真正有用的新信号；
2. `Math-PRM-7B` 冻结 backbone 是目前最重要的 backbone breakthrough；
3. benchmark-oriented hybrid（`PRMBench local + bounded terminal + small broader-prefix support`）
   是当前最强的 E 阶段路线；
4. F 阶段最有前景的是 `heuristic / BC controller`，而不是从零开始 RL。

### 7.3 诚实的当前边界

1. verifier 很强，但还不能直接宣传成 fully RL-ready；
2. terminal completion ordering 仍是关键残差；
3. live controller 验证还没正式完成；
4. answer-verifier / abstention / process-outcome alignment 仍待补全。

## Addendum (2026-03-12 late follow-up): `PH1/PH2` controller distillation audit

在写作准备期间，我们又补做了一轮更聚焦的 `Phase F` 复核，目标是回答一个具体问题：

1. 先前 `PH2-MLP-Math` 上那次看似积极的 `BC->RL` 结果，到底是不是稳定信号？

这轮复核的方法更严格：

1. 不再使用 wrapper 里的 convenience teacher；
2. 改用该 slice 在 controller sweep 中的真实最佳 heuristic teacher；
3. 做 `3` 个 seed；
4. 同时保留旧 teacher 作为 control。

结果非常关键：

| case | teacher | mode | mean test balanced_f1 | std |
|---|---|---|---:|---:|
| `ph2_mlp_math` | `guarded_drop` | `bc_only` | `0.7292` | `0.1289` |
| `ph2_mlp_math` | `guarded_drop` | `bc_then_rl` | `0.6929` | `0.0815` |
| `ph2_mlp_math` | `threshold_only` | `bc_only` | `0.8020` | `0.0273` |
| `ph1_gated_math` | `drop_needs_low` | `bc_only` | `0.8030` | `0.0690` |
| `ph1_gated_math` | `drop_needs_low` | `bc_then_rl` | `0.8211` | `0.0490` |

新结论：

1. `PH2-MLP-Math` 上先前那次 `BC->RL` 小幅增益，不能再被当作稳健证据；
2. `BC->RL` 当前仍然是 slice-specific optional follow-up，而不是默认主线；
3. 当前最稳妥的 `Phase F` 叙述应继续保持：
   - heuristic family selection
   - BC distillation
   - selective BC->RL
   - not from-scratch RL
