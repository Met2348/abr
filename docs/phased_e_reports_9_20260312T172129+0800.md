## 3. Phase D: what was tried, what was learned, and what failed

### 3.1 Why Phase D existed

Phase D answered one narrow but crucial question:

1. if we train on higher-quality external step/pair data,
2. can the value head learn meaningful ordering at all?

This mattered because earlier StrategyQA-coupled lines could not separate “training objective is wrong” from “the task itself is unlearnable under the current setup.”

### 3.2 What Phase D successfully established

The most important surviving artifact is:

1. `assets/artifacts/phase_d_gate_reports/post_dt2_stable/gate_report.json`

Key result:

1. `d6t.pass = true`
2. `DT2_MATH_SHEPHERD_SEED3_STABLE`
   - `mean_pair_acc = 0.8154`
   - `mean_auc = 0.7857`
   - low cross-seed variance
3. `DT4_MIXED_MS_PRM800K_SEED3_STABLE`
   - `mean_pair_acc = 0.6612`
   - `mean_auc = 0.6616`
   - also stable enough to pass the gate

Interpretation:

1. high-quality external pair/triplet supervision is genuinely learnable,
2. the basic ranking/value training pipeline is not fundamentally broken.

### 3.3 What Phase D did not solve

The same gate report shows:

1. `d6.pass = false`
2. current blocker recorded there is `missing_d6_pair_metrics`
3. earlier historical reading already showed StrategyQA-side behavior near random or not decision-complete

The surviving Phase D conclusion is therefore:

1. external learnability is real,
2. target-domain transfer is the hard problem,
3. the main bottleneck is not “trainer cannot learn any ranking signal.”

### 3.4 How Phase D changed the project

Phase D should now be interpreted as the phase that killed one bad assumption:

1. “If the value head underperforms on the original task, the whole approach is probably dead.”

That assumption is no longer defensible.

The stronger conclusion is:

1. there is learnable value/reward signal,
2. but source-target mismatch is severe,
3. therefore Phase E needed a benchmark-native pivot.

## 4. Phase E pivot: from StrategyQA coupling to benchmark-native verification

### 4.1 Why the pivot was necessary

After Phase D, the repo had enough evidence that the right question was no longer:

1. “Can we salvage StrategyQA with more tricks?”

The right question became:

1. “Can we train a verifier/value-like head that is genuinely useful on a benchmark designed for step-level reasoning trust?”

This is why Phase E moved to:

1. Math-Shepherd
2. PRMBench / PRMBench Preview
3. ProcessBench
4. DPO-like math pair data

### 4.2 Phase E early result: same-source Math-Shepherd is learnable

This has held up across many later audits.

The important positive result is:

1. same-source / same-family Math-Shepherd supervision can train a useful verifier

This is the foundation for everything after it:

1. it justifies using step/pair data as the main supervised object,
2. it justifies discussing Phase F at all,
3. it also sets an upper bound on what “single-source success” means.

It does **not** prove broad generalization.

## 5. Early Phase E failure modes that still matter

### 5.1 Multi-source Stage-B failure was real and diagnostic

Earlier Stage-B mixed-source runs failed in a way that is now well-understood.

The core issues were:

1. mixing incompatible supervision semantics as if they were one task,
2. insufficient optimizer-step budget in some mixed runs,
3. confidence/source weighting being disabled in exactly the regimes that needed it,
4. benchmark-facing metrics not matching same-source held-out success,
5. `sigmoid`-space ranking geometry being too easy to saturate in earlier recipes.

This line was important because it showed that:

1. “more heterogeneous data” is not automatically better,
2. `pair_acc` can improve while ProcessBench remains weak,
3. architecture is not the first thing to blame when data geometry is wrong.

### 5.2 R-PRM ingestion had a concrete engineering bug

This was fixed earlier in the project and still matters historically.

The problem:

1. the old `R-PRM` contract fed long verifier essays into a frozen-feature scalar head pipeline,
2. which made truncation and signal dilution a first-order failure mode.

The repair:

1. compact verdict contracts were added,
2. direct long-essay paths were no longer used as the safe default.

Impact:

1. this removed one false negative source,
2. but `R-PRM` still did not become the main winning direction.

### 5.3 Judge-as-bulk-filter failed

Two lines of evidence are now strong enough to treat this as a negative result:

1. `PRMBench_Preview selected relabel`
2. `ProcessBench hard-slice benchmark-side adjudication`

Artifacts:

1. `assets/artifacts/phase_e_logs/phase_e_prmbench_selected_relabel_formal_0311/final_summary.md`
2. `assets/artifacts/phase_e_logs/phase_e_processbench_hardslice_adj_0311_r2/final_summary.md`

Key numbers:

1. `selected relabel`
   - baseline `pair_acc=0.9521`, `auc=0.9071`
   - selected relabel `pair_acc=0.9555`, `auc=0.9207`
   - but judge keep rate on the hard slice was only `0.0156`
2. `ProcessBench hard-slice adjudication`
   - `math both_parse_ok = 0.0345`
   - `gsm both_parse_ok = 0.0370`
   - benchmark-side reliable adjudication essentially failed

Interpretation:

1. the apparent PRMBench gain was mostly “delete hard/noisy examples,” not trusted semantic relabeling,
2. the local judge is currently useful only for offline audit or extreme pruning,
3. it is not a trustworthy bulk relabeler and not a trustworthy ProcessBench adjudicator.

## 6. ProcessBench transfer repair line: what it taught us

### 6.1 Why this line mattered

The hardest open issue in Phase E has never been same-source held-out ranking.

It has been:

1. first-error detection,
2. later-bad discrimination,
3. all-correct completion handling,
4. benchmark-facing stability.

That is why the transfer-repair matrix around `E79-E90` was important.

### 6.2 The main surviving conclusion from the transfer-repair line

The repo already established this before the current pass, and it still holds:

1. adding terminal anchors helps address one real blind spot,
2. but over-weighting terminal supervision damages the ranking surface,
3. low-mass terminal augmentation is safer than heavy terminal augmentation,
4. none of the current scalar recipes is yet strong enough to call `RL-ready`.

This matters because it is exactly the kind of subtle failure that literature also warns about:

1. reward/value signal can improve one slice while breaking another,
2. and RL will amplify whichever failure mode remains.

## 9. PBR38B: the most important recent negative result

### 9.1 Original hypothesis

`PBR38B` tried to test a good literature-backed idea:

1. consensus filtering,
2. better quality should beat more quantity.

The hypothesis was reasonable because Qwen-style and related pipelines often benefit from agreement filtering.

### 9.2 Actual result

`PBR38B` held-out run:

1. `pair_acc = 0.5481`
2. `auc = 0.5253`

From:

1. `assets/artifacts/phase_e_runs/phase_e_pbr38b_consensus820_frozen_mlp_s42_20260312T085318Z/eval_metrics.json`

This is not a mild underperformance.

It is collapse-level behavior relative to the current frontier.

### 9.3 Root cause uncovered in this pass

This pass added an explicit pair-pool audit suite and confirmed the actual failure mode.

Artifact:

1. `assets/artifacts/phase_e_analysis/phase_e_current_frontier_audit_pair_pools_20260312T091853Z/summary.json`

Important facts:

1. `PBR38B` train pool size shrank to `2015`
2. its train semantics became:
   - `sibling_branch = 1561`
   - `terminal_completion_anchor = 454`
   - `local_first_bad_edge = 0`
3. `terminal_fraction = 0.225`

This means `PBR38B` was not “consensus-filtered PBR26” in any benchmark-safe sense.

It was effectively:

1. DPO sibling-heavy,
2. terminal-heavy,
3. with local first-bad supervision deleted.

That directly explains why it collapsed.

### 9.4 Why this matters beyond one run

This is one of the most useful recent negative results in the whole repo.

It proves:

1. naive consensus filtering can be structurally wrong,
2. quality filtering must preserve benchmark-relevant semantics,
3. “better labels” is not meaningful if the filtering rule deletes the supervision geometry the benchmark actually tests.

## 12. Literature-backed redesign judgment after this pass

### 12.1 What the literature now says clearly

The literature no longer supports the idea that the end-state object should be a single scalar head trusted everywhere.

The strongest cross-paper consensus is:

1. process supervision is useful,
2. but it must align with benchmark geometry,
3. richer critics/verifier systems are increasingly favored,
4. selective escalation is part of the system design,
5. RL is more sensitive to reward/verifier quality than offline reranking.

### 12.2 What that means for this repo

The repo should now distinguish three objects:

1. `local/process verifier`
   - should detect first-bad and later-bad structure,
2. `terminal/answer verifier`
   - should detect completion acceptability / all-correct behavior,
3. `abstain/escalate logic`
   - should decide whether cheap verification is sufficient.

The current negative dual-head result does **not** invalidate this design.

It only invalidates the current local implementation of it.

### 12.3 Why single-scalar iteration is no longer enough

Current repo evidence now matches literature on three points:

1. same-source success does not imply benchmark trust,
2. broader plasticity without better data can hurt,
3. process-outcome alignment slices are exactly where naive reward/value formulations break.

Therefore the project should stop asking:

1. “Can we push one scalar head a little more?”

and instead ask:

1. “What is the minimal verifier system and data contract that preserves local reasoning trust, terminal completion trust, and safe escalation?”

## 13. Updated data pipeline design

### 13.1 Data pipeline principles

The next training-data mainline should follow these rules:

1. split by `source_sample`, never by raw pair only.
2. cap with `balanced_support_bucket`, not file order.
3. preserve explicit semantic coverage:
   - local first-bad
   - sibling/fork-point
   - bounded terminal anchors
4. consensus filtering must be semantics-aware.
5. judge-assisted filtering should remain optional and conservative.

### 13.2 Concrete data lines to keep

Keep:

1. strict or higher-quality Math-Shepherd local supervision,
2. DPO sibling/fork-point data,
3. small bounded terminal anchors,
4. PRMBench / ProcessBench only for audit or carefully aligned auxiliary uses.

Do not promote:

1. naive bulk judge relabel,
2. naive consensus filters that erase local supervision,
3. heavy terminal-dominated pools.

### 13.3 Concrete new candidate after this pass

The new candidate pair pool is:

1. `PBR40` semantic-consensus v1

This should be the next direct data baseline to train before inventing another large set of new tricks.

## 14. Updated training pipeline design

### 14.1 Short-term training priorities

Short-term mainline should be:

1. frozen baseline on `PBR40`
2. all-layer LoRA on `PBR40`
3. benchmark-facing evaluation with fixed-threshold `ProcessBench F1`
4. same-family eval as mandatory, not optional

### 14.2 What not to over-invest in immediately

Do not prioritize:

1. more rank sweeps on noisy `PBR26`-like pools,
2. more judge prompt tuning for bulk relabel,
3. new Stage-B-style mixed semantics without explicit contract separation.

### 14.3 More literature-aligned training targets worth trying

The most promising next-label / next-loss directions are:

1. `PAV`-style progress targets
2. `RPE`-style soft progress labels
3. semantic-aware weighting instead of global mixing
4. explicit local-vs-terminal factorization

These should be attempted only after the safer curation contract is in place.

## 15. Architecture redesign judgment

### 15.1 Current architecture lesson

The project has already run enough evidence to justify:

1. frozen scalar head is a valid baseline,
2. LoRA can help when the data is clean enough,
3. capacity alone does not solve poor supervision geometry.

### 15.2 Architecture hierarchy going forward

Recommended order:

1. `frozen mlp / gated_mlp` on well-curated data
2. `all-layer LoRA + small head` on well-curated data
3. `process head + terminal head + downstream fusion`
4. only after that, revisit more ambitious critic/generative verifier lines

### 15.3 What the repo should not claim today

The repo should not currently claim:

1. one universal scalar head is enough,
2. all-layer LoRA has beaten frozen for the broad noisy pool,
3. current verifier stack is RL-ready.

## 16. Phase F groundwork: what is already solid and what is not

### 16.1 What is solid enough

The following Phase F groundwork conclusions have held up:

1. `heuristic -> BC -> selective BC->RL` remains the correct controller progression.
2. from-scratch RL-like controller training is still unstable.
3. benchmark saturation makes `GSM8K` a poor main RL proving ground.
4. reward shaping must be bounded and delta-like, not raw-score maximizing.

### 16.2 What is still not solid enough

Not yet solid:

1. a fully trustworthy Math verifier for cheap-to-strong escalation,
2. a robust ProcessBench-hard-slice adjudicator,
3. a benchmark-proven implicit PRM line,
4. a generally promotable online RL result.
