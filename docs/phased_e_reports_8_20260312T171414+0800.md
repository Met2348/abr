## 1. Stable Conclusions First

The most stable project conclusions at this point are:

1. `Phase D` proved that ranking-style external process supervision is learnable, but only ranking-first bridge training helps the downstream corruption-ranking target.
2. `Phase E` proved that same-source fit is easy to over-read. Same-source `pair_acc > 0.93` is not equivalent to benchmark-facing verifier quality.
3. the strongest current `Phase E` lines are no longer `R-PRM` or bulk source concat. The frontier is now:
   - `PBR26/PBR32` style math verifier lines for the older scalar-head track;
   - `PH1/PH2/PH3` style hybrid verifier lines for the newer system-oriented track.
4. `Phase F` keeps repeating the same message under stronger audits: `heuristic / BC first, RL second`.
5. the repo had several high-risk silent-failure and silent-misread pitfalls. The important ones are now known and mostly fixed.
6. the correct long-term object is not “one scalar head that solves everything”; it is a verifier system:
   - cheap verifier,
   - stronger verifier or judge on hard slices,
   - abstain/escalate logic,
   - controller/reranker interfaces,
   - only then RL promotion.

## 2. Short Glossary For Writing / Reporting

1. `PRM`: process reward model, i.e. step/prefix quality scorer.
2. `ORM` / outcome reward: reward based only on final answer correctness.
3. `BiRM` / terminal BCE: auxiliary terminal supervision that anchors the final step score.
4. `BoN`: Best-of-N reranking.
5. `BC`: behavior cloning.
6. `BC->RL`: behavior cloning warm start followed by RL-style fine-tuning.
7. `GRPO`: Group Relative Policy Optimization.
8. `DAPO`: a 2025 GRPO-family recipe emphasizing asymmetric clipping and training stability.
9. `Dr. GRPO`: a debiased GRPO variant that removes problematic per-group variance normalization effects.
10. `cheap -> strong gate`: use a cheap verifier on easy slices and escalate only ambiguous cases to a stronger verifier.

## 4. Phase E Part I: Same-Source Learnability And Contract Diagnosis

### 4.1 ACC90-style same-source results

### 4.2 `R-PRM` diagnosis

Representative artifacts:

1. `assets/artifacts/phase_e_runs/phase_e_rprm_trainfit_probe_0310_s4k_20260310T160730Z/summary.md`
2. `assets/artifacts/phase_e_runs/phase_e_rprm_heldout_repair_0310_s4k_20260310T164443Z/summary.md`

Takeaway:

1. train fit can exceed `0.90` pair accuracy;
2. true held-out remains weak (`~0.63` pair accuracy);
3. under the current compact contract, `R-PRM` is not trustworthy enough to be the main anchor.

### 4.3 Anti-patterns that mattered

The repo repeatedly rediscovered these failure modes:

1. dangerous recipe families such as `logit + confidence_semantic + ranking_score` can look superficially normal while destroying benchmark behavior;
2. `Math-Shepherd` geometry can inject depth/length bias if the pair contract is not strict enough;
3. terminal behavior and local first-bad discrimination are different skills and must not be conflated.

These diagnoses are why later Phase E trainers defaulted back to repository-safe settings such as `score + none + pair_acc`.

## 10. Literature-Backed Redesign Direction

This direction is consistent with both the repo’s experiments and the web-verified primary sources already mirrored under `docs/relatedPapers/`.

### 10.1 Data curate redesign

The data pipeline should prioritize benchmark-facing geometry in this order:

1. local first-bad anchor pairs from the strongest benchmark-aligned source;
2. bounded terminal/all-correct auxiliary supervision;
3. small broader-prefix support such as grid/sibling support, but only in measured quantities;
4. selected-relabel or judge repair only on ambiguous low-margin slices, never as a blind bulk filter;
5. strict provenance recording so every pair family can be ablated independently.

### 10.2 Model / system redesign

1. keep a cheap verifier head for dense prefix scoring;
2. keep architecture variants such as `mlp`, `gated_mlp`, and ensemble mixes rather than forcing one winner too early;
3. add selective escalation rather than trying to make one model perfect at every slice;
4. treat controller/reranker value as a first-class deployment target, not only as a pre-RL stepping stone.

### 10.3 RL redesign

1. treat GSM8K live RL as infrastructure canary only.
2. put stronger weight on modern GRPO stabilization knobs:
   - `clip_delta` process reward shaping,
   - `dapo` / `dr_grpo` family losses,
   - explicit `beta`, `scale_rewards`, truncation masking,
   - disk-safe checkpoint policy.
3. keep process-reward RL behind preflight gates:
   - threshold stability,
   - reward-hacking probes,
   - controller usefulness,
   - harder benchmark selection.
